# MyPort 설계 문서

## 1. 문서 개요

- 문서명: MyPort 설계 문서
- 버전: v0.4
- 작성일: 2026-03-25
- 대상: 개인용 iOS 자산 포트폴리오 관리 앱
- 변경 사항: `서버 저장 중심 구조`에 `Apple Vision OCR 분석 파이프라인`, `히스토리/변화 분석/자산 비중 시각화` 설계 추가

## 2. 제품 요약

MyPort는 사용자가 은행, 증권사, 코인 거래소, 기타 자산 앱의 화면을 캡처해 업로드하면, 서버가 이미지를 분석하고 자산군별로 분류하여 포트폴리오 스냅샷을 저장하는 개인용 iOS 앱이다.

핵심 방향은 다음과 같다.

- 금융사 API 연동 없이 스크린샷 기반으로 동작한다.
- 포트폴리오 원본 데이터는 서버에 저장한다.
- 앱은 서버의 최신 스냅샷을 조회하고 수정 결과를 다시 서버로 저장한다.
- 스크린샷 원본 이미지와 분석 결과는 서버에서 함께 관리한다.
- 앱에는 최소한의 캐시와 설정만 둔다.

## 3. 이번 설계 변경의 핵심

기존 설계:

- 앱 내부 `SwiftData`가 원본 저장소
- 이미지와 분석 결과를 기기 내부에 저장

변경 설계:

- 서버 DB와 오브젝트 스토리지가 원본 저장소
- 앱은 `서버 API 클라이언트` 역할 수행
- 분석 결과와 환율, 원본 이미지 메타데이터를 서버에 저장
- 앱은 필요 시 최신 스냅샷을 동기화해 내려받음

이 변경으로 얻는 장점:

- 기기 교체 시 데이터 유지 가능
- 향후 웹/태블릿/관리 콘솔 확장 가능
- 이미지 분석을 서버에서 일관되게 처리 가능
- 대용량 이미지와 분석 이력 관리가 쉬워짐

주의할 점:

- 서버 인증과 보안 설계가 필요함
- 업로드/분석/저장 파이프라인 운영 비용이 생김
- 네트워크 실패, 동기화 지연, 부분 저장 실패를 처리해야 함

## 4. 제품 목표

### 4.1 목표

- 여러 금융 앱 스크린샷에서 자산 정보를 자동 추출한다.
- 자산을 아래 자산군으로 자동 분류한다.
  - 국내주식
  - 해외주식
  - 현금성자산
  - 코인
  - 채권
- 자산군별 합계와 종목별 상세를 한 화면에서 확인할 수 있게 한다.
- 스냅샷 생성 시점의 환율을 함께 저장한다.
- 모든 자산을 원화 기준 총합으로 계산해 보여준다.
- 서버에 저장된 데이터를 기기에서 조회, 수정, 재분석할 수 있게 한다.
- 날짜별 스냅샷 히스토리를 모두 확인할 수 있게 한다.
- 이전 기록 대비 총자산, 자산군, 종목 단위 증감을 확인할 수 있게 한다.
- 특정 스냅샷 시점의 자산 비중을 원형그래프로 확인할 수 있게 한다.

### 4.2 비목표

- 금융기관 API 직접 연동
- 자동 매매
- 멀티유저 협업 기능
- 초기 버전에서 모든 금융 앱 UI 지원

## 6. 사용자 시나리오

### 6.1 신규 업로드

1. 사용자가 앱에서 여러 장의 스크린샷을 선택한다.
2. 앱이 서버에 업로드 세션을 생성한다.
3. 앱이 이미지를 업로드한다.
4. 서버가 OCR, 파싱, 분류, 환율 연결, 원화 합산을 수행한다.
5. 서버가 스냅샷을 저장한다.
6. 앱이 저장된 스냅샷을 다시 조회해 보여준다.

### 6.2 수동 기록

1. 사용자가 자산과 환율을 직접 입력한다.
2. 앱이 스냅샷 생성 요청을 서버로 보낸다.
3. 서버가 저장 후 응답한다.
4. 앱이 최신 목록을 새로고침한다.

### 6.3 기존 스냅샷 확인

1. 사용자가 앱을 연다.
2. 앱이 서버에서 최신 스냅샷 목록을 가져온다.
3. 사용자가 스냅샷 상세를 열어 자산군별 합계, 통화별 합계, 기록 당시 환율, 원화 총합을 확인한다.

### 6.4 날짜별 히스토리 확인

1. 사용자가 `히스토리` 화면을 연다.
2. 앱이 저장된 스냅샷을 기록일 기준 내림차순으로 보여준다.
3. 사용자는 월별로 기록된 모든 스냅샷을 확인할 수 있다.
4. 각 행에서 기록일, 총자산 원화, 직전 대비 증감, 주요 비중 정보를 확인한다.

### 6.5 이전 기록 대비 변화 확인

1. 사용자가 특정 스냅샷 상세를 연다.
2. 앱이 가장 가까운 이전 스냅샷을 찾는다.
3. 앱이 총자산 변화, 자산군 변화, 종목별 증가/감소/신규/소멸을 계산한다.
4. 사용자는 예를 들어 `국내주식 +1,200,000원`, `코인 -840,000원`, `Apple +350,000원`처럼 확인한다.

### 6.6 자산 비중 원형그래프 확인

1. 사용자가 스냅샷 상세 또는 분석 탭을 연다.
2. 앱이 해당 스냅샷의 자산군별 원화 합계를 계산한다.
3. 앱이 `국내주식 / 해외주식 / 코인 / 현금성자산 / 채권` 비중을 원형그래프로 그린다.
4. 사용자는 각 비중의 퍼센트와 원화 금액을 함께 본다.

## 8. 권장 기술 방향

이번 설계 문서는 특정 SaaS에 종속되지 않도록 `범용 REST API + 오브젝트 스토리지` 구조를 기준으로 한다.

### 8.1 iOS

- Swift
- SwiftUI
- URLSession
- PhotosPicker
- Vision
- Charts

### 8.2 서버

- REST API
- 비동기 작업 큐
- JSON 기반 요청/응답
- 파일 업로드 API 또는 Presigned URL

### 8.3 저장소

- 스냅샷 메타데이터: DB
- 원본 이미지: object storage
- 인증 토큰: iOS Keychain
- 앱 설정 및 임시 캐시: UserDefaults/File Cache

## 9. 서버 저장 기준 데이터 흐름

### 9.1 수동 입력 스냅샷 저장

1. 사용자가 환율과 보유 자산을 입력
2. 앱이 `POST /v1/snapshots` 호출
3. 서버가 유효성 검사
4. 서버가 DB 저장
5. 서버가 저장된 스냅샷 반환
6. 앱이 UI 반영

### 9.2 이미지 업로드 기반 분석 저장

1. 앱이 업로드 세션 생성 요청
2. 서버가 업로드 ID와 업로드 경로 반환
3. 앱이 이미지 업로드
4. 앱이 분석 시작 요청
5. 워커가 OCR 및 분석 수행
6. 서버가 스냅샷 저장
7. 앱이 분석 상태 폴링 또는 웹소켓으로 결과 수신

### 9.3 조회 흐름

1. 앱 실행
2. 인증 토큰 확인
3. `GET /v1/snapshots`
4. 최신 데이터 표시
5. 필요 시 상세 재조회

### 9.4 히스토리 및 비교 분석 흐름

1. 앱이 스냅샷 목록을 모두 또는 기간 기준으로 조회
2. 앱이 `capturedAt` 기준으로 정렬
3. 앱이 현재 스냅샷과 직전 스냅샷을 매칭
4. 앱이 총합, 자산군, 종목 레벨 차이를 계산
5. 앱이 차트 및 리스트 형태로 표시

## 10. 권한 및 보안 설계

### 10.1 인증

MVP 기준으로는 다음 두 방식 중 하나를 선택한다.

- 개인 서버용 Bearer Token
- 단일 사용자 이메일 로그인 후 JWT

초기 구현은 가장 단순한 `Bearer Token` 기반이 적합하다.

### 10.2 보안 원칙

- 토큰은 Keychain 저장
- Base URL, mock 모드 등 비민감 설정만 UserDefaults 저장
- 원본 이미지 업로드는 HTTPS 강제
- 서버는 사용자별 데이터 접근 통제
- 이미지와 분석 결과에 대한 접근 로그를 남길 수 있는 구조 권장

### 10.3 앱 단 보안

- 앱 잠금 기능 유지
- 민감 화면 블러 처리
- 토큰 노출 방지

## 13. 환율 및 원화 총합 설계

### 13.1 원칙

- 환율은 반드시 스냅샷 시점 기준으로 저장한다.
- 현재 환율이 아닌 `기록 당시 환율`을 기준으로 원화 총합을 계산한다.
- 통화별 합계와 원화 환산 합계를 동시에 보여준다.

### 13.2 계산 규칙

- `KRW`는 환율 1로 처리
- 외화 및 스테이블코인은 스냅샷에 저장된 환율 사용
- 환율이 없는 통화는 합계 계산에서 경고 대상으로 표시

### 13.3 예시

- `USD 100`과 `USD/KRW 1472.30`이면 원화 환산은 `147,230`
- `USDT 50`과 `USDT/KRW 1471.80`이면 원화 환산은 `73,590`

## 14. 히스토리 설계

### 14.1 기본 원칙

- 사용자가 기록한 모든 스냅샷은 보존한다.
- 히스토리 정렬 기준은 `capturedAt` 내림차순이다.
- 월 1회 기록이 기본 사용 패턴이지만, 더 자주 기록해도 지원한다.
- 같은 달에 여러 기록이 있으면 모두 표시하고, 필요 시 `월별 대표값`은 해당 월의 최신 스냅샷으로 계산한다.

### 14.2 히스토리 화면 구성

- 상단 요약
  - 최근 총자산
  - 최근 기록일
  - 직전 대비 증감
- 필터
  - 전체
  - 최근 3개월
  - 최근 6개월
  - 최근 1년
- 리스트
  - 기록일
  - 총자산 KRW
  - 직전 대비 증감
  - 주요 자산군 비중

### 14.3 월 라벨 규칙

- `2026-03-25` 기록은 `2026년 3월` 그룹에 속한다.
- 히스토리 화면은 기본적으로 월 단위 섹션 헤더를 사용한다.
- 섹션 안에는 실제 기록 날짜별 항목을 나열한다.

## 15. 변화 분석 설계

### 15.1 기준 스냅샷 선택

- 기본 비교 기준은 `직전 스냅샷`이다.
- 사용자가 원하면 임의의 과거 스냅샷과도 비교할 수 있도록 확장 가능하다.

### 15.2 비교 단위

- 총자산 KRW 변화
- 자산군별 KRW 변화
- 종목별 KRW 변화
- 신규 편입 / 전량 매도 / 유지 상태

### 15.3 종목 매칭 규칙

종목 비교의 키는 다음 우선순위로 만든다.

1. `symbol + institution + assetClass + currency`
2. 심볼이 없으면 `name + institution + assetClass + currency`

이 규칙으로 현재 스냅샷과 이전 스냅샷의 종목을 매칭한다.

### 15.4 변화 계산 규칙

- `deltaKRW = currentValueKRW - baselineValueKRW`
- `deltaPercent = deltaKRW / baselineValueKRW`
- 기준값이 0이면 퍼센트는 표시하지 않거나 `신규`로 처리
- 현재만 존재하면 `new`
- 과거에만 존재하면 `removed`
- 차이가 거의 없으면 `unchanged`

### 15.5 보여줄 항목

- 총자산 변화 카드
- 자산군 변화 리스트
- 상승 상위 종목 5개
- 하락 상위 종목 5개
- 신규 편입 종목
- 사라진 종목

## 16. 자산 비중 원형그래프 설계

### 16.1 데이터 소스

- 특정 스냅샷의 자산군별 `totalKRW`
- 분류 대상:
  - 국내주식
  - 해외주식
  - 코인
  - 현금성자산
  - 채권
  - 필요 시 미분류

### 16.2 계산 규칙

- 각 자산군의 원화 총합을 전체 총자산 KRW로 나눈다.
- 비중이 0이거나 항목이 없는 자산군은 차트에서 생략 가능하다.
- 최소 표시 비율 미만 항목은 `기타`로 합칠 수 있지만 MVP에서는 개별 표시를 우선한다.

### 16.3 UI 규칙

- 차트 위 또는 중앙에 총자산 KRW 표시
- 차트 아래 범례에 자산군명, 금액, 퍼센트 표시
- 자산군 색상은 앱 전반의 `AssetClass.tint`와 일치

### 16.4 기술 방향

- iOS 18 기준 `Swift Charts`의 `SectorMark` 사용
- Charts 미사용 환경으로 확장 시 커스텀 `Path` 렌더링으로 대체 가능

## 17. 서버 API 설계 원칙

### 17.1 MVP 전략

- 히스토리/변화 분석/비중 차트의 원본 데이터는 기존 `GET /v1/snapshots` 응답으로 충분하다.
- MVP에서는 앱이 스냅샷 목록을 내려받아 로컬에서 파생 계산한다.
- 서버는 원본 스냅샷 저장과 조회에 집중한다.

### 17.2 선택적 확장 API

향후 스냅샷 수가 많아지면 다음 API를 추가할 수 있다.

- `GET /v1/snapshots?from=...&to=...&limit=...`
- `GET /v1/analytics/comparison?snapshotId=...&baselineSnapshotId=...`
- `GET /v1/analytics/allocation?snapshotId=...`
- `GET /v1/analytics/timeline?groupBy=month`
### 17.3 엔드포인트 범위

- 스냅샷 목록 조회
- 스냅샷 상세 조회
- 스냅샷 생성
- 스냅샷 삭제
- 이미지 업로드 세션 생성
- 분석 작업 상태 조회

### 17.4 응답 원칙

- 날짜는 ISO 8601
- 금액과 환율은 JSON number
- 자산과 환율은 스냅샷 응답에 포함

## 25. 결론

MyPort는 이제 `로컬 전용 포트폴리오 앱`이 아니라 `서버 저장형 스크린샷 기반 포트폴리오 앱`으로 설계가 전환되었다. 따라서 iOS 앱의 핵심 책임은 `업로드/조회/편집 클라이언트`, 서버의 핵심 책임은 `원본 데이터 저장과 분석 처리`가 된다.

이번 설계 반영의 핵심은 다음 다섯 가지다.

- 스냅샷과 환율의 원본 저장소를 서버로 이동
- 앱 구조를 Repository 기반 원격 저장소 중심으로 변경
- 기록 당시 환율과 원화 총합 계산 로직은 유지하되 서버 응답 기반으로 동작하게 변경
- 과거 스냅샷을 전부 보존하는 히스토리 구조 채택
- 변화 분석과 자산 비중 차트는 앱에서 파생 계산하는 방식 채택
